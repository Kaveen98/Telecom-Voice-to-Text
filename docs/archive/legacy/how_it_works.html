🎙 SLT Transcription System — How It Works

From audio file → transcript + cost in LKR

📁
Step 1 — You drop an audio file
Drop any .mp3 / .wav / .m4a file into the input_audio/ folder. That's literally the only thing you do. Everything else is automatic.
input_audio/call.mp3
👀
Step 2 — watcher.py detects the file
watcher.py is always watching the folder using Linux's built-in file notification system. The moment a new file appears, it adds it to a queue and starts processing.
watcher.py
🔧
Step 3 — ffmpeg cleans the audio
Before sending to Google, ffmpeg (a free audio tool) does two things:
1. Converts the file to a standard format (16kHz mono WAV)
2. Strips silence — removes quiet gaps to save token costs (silence is billed the same as speech, so removing it saves money)
ffmpeg → converted + silence stripped
☁️
Step 4 — Audio sent to Google Vertex AI
The cleaned audio is sent to Google's Gemini model (whichever model is set in MODEL_NAME). Google listens to it and returns:
• The transcript text (with language tags [SI] [EN] [TA])
• Exact token counts: how many tokens for input, output, and thinking
gemini_flash_stt.py → Vertex AI API
💰
Step 5 — Cost calculated in USD and LKR
Using the token counts Google returned, the code multiplies by the model's price per million tokens. Thinking tokens are included in output cost because Google bills them that way. Final cost shown in both USD and LKR.
tokens × price/1M = cost → × 305 = LKR
💾
Step 6 — Everything saved automatically
Two things are saved at the same time:
1. Transcript text → saved to output/call_transcript.txt
2. Full call record (tokens, cost USD, cost LKR, languages, duration) → saved to calls.db database for the dashboard
database.py → calls.db  |  output/*.txt
📊
Step 7 — Dashboard updates
The dashboard at http://localhost:5050 reads from calls.db and refreshes every 10 seconds. You can see every call's cost in LKR, language breakdown, and running totals for the day.
dashboard_server.py → http://localhost:5050
